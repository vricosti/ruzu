## Session findings: Phase 1/2 are unworkable, plan revised

Four separate tactical fixes to the lock model were attempted this session.
All four regressed boot (details in
`~/.claude/.../memory/project_mk8d_lock_refactor_attempts_all_regressed.md`):

| Change | Direction | Baseline | Result |
|---|---|---:|---:|
| preempt clone-before-yield | remove sync | 53 s | 3 s |
| access_unlocked pilot | switch primitive at one site | 53 s | 23 s |
| handle_interrupt thread→process restructure | remove sync | 53 s | 4 s |
| handle_interrupt add KScopedSchedulerLock (Phase 1a) | **ADD** sync, upstream-faithful | 53 s | **1 s / 0 audio** |

The fourth is the telling one: **adding** scheduler-lock coverage at a
single site (exactly as upstream does it) regressed the hardest. This
proves the incremental Phase 1 / Phase 2 approach cannot work; the
codebase's current equilibrium is too tangled for single-site changes
in either direction.

## Revised plan: upstream translation + environment prerequisites

The audit's "categories A–F" framing was premature procrastination. The
correct framing is: **port upstream directly, but first fix the ruzu
subsystems that upstream's code assumes are true.**

Upstream `SignalToAddress` starts with:
```cpp
KThread* owner_thread = GetCurrentThreadPointer(kernel);  // always valid
```

Ruzu's `get_current_emu_thread()` returns `Option<Arc<Mutex<KThread>>>`
and frequently returns `None` at init / fiber-transition moments. Phase 1a
failed exactly there: `scheduler_lock_ptr` was 0 at the interrupt
moment because the current thread hadn't been populated yet, so the
upstream-faithful code took a silent no-lock fallback and raced.

**Corrected phase ordering:**

1. **Make ruzu's current-thread accessor match upstream's totality** —
   `get_current_emu_thread()` must always return a valid KThread during
   CPU execution (fall back to an idle thread when nothing else is
   scheduled, matching upstream). This is its own session: touches
   thread-local population, fiber-switch transitions, HLE handlers.
2. Translate condvar / arbiter / wait paths directly from upstream —
   now that `scheduler_lock_ptr` is always valid, `KScopedSchedulerLock`
   and `KScopedSchedulerLockAndSleep` ports work as intended.
3. Do the atomic type swap: `Arc<Mutex<KProcess>>` → `Arc<KProcessCell>`,
   `Arc<Mutex<KThread>>` → `Arc<KThreadCell>`. Compiler enforces
   atomicity; every `.lock().unwrap()` call site must be rewritten.
4. Remove `TrackedMutex` and the pre-existing diagnostic scaffolding.

Each step is its own commit, each testable. Step 1 is the prerequisite
that this session discovered was missing — Phase 1a's regression was
rooted there, not in the lock changes themselves.

**Do NOT attempt Phases 1/2 from the original plan** (wrap SVC bodies,
HLE migration). Both proven to regress when applied incrementally.

## Progress — 2026-04-20

### Landed

1. ✅ **Step 1 (current-thread accessor totality)** — `ae42dfc` made
   `get_current_thread_id_fast` / `with_current_thread_fast_mut` total.
2. ✅ **Step 3 (atomic type swaps)** — `d0f45e4` (`ProcessLock =
   SyncCell<KProcess>`) + `3514834` (`KThreadLock = SyncCell<KThread>`).
   Both are "zero call-site changes" — SyncCell exposes API-compatible
   `.lock()` / `.try_lock()` / `.lock_with()` / `.from_value()` shims.
3. ✅ **SIGUSR1 investigation overlay** — `ee5bbd3` squash-merged the
   TrackedMutex-based debugger while preserving the SyncCell direction
   (ProcessLock stays `sync_cell::KProcessCell`; TrackedMutex unused).
4. ✅ **Step 2 — condvar + svc_lock**: `a28cbe4` landed
   `kernel::scheduler_lock()` singleton accessor, ripped out the
   "scheduler_lock_ptr == 0 → silently no-op" fallback across
   `k_condition_variable.rs`, and tightened `wait_for_address` to drop
   the scheduler lock BEFORE the fiber-wait begins (holding it across
   the wait deadlocks EndWait on the signaling thread — discovered by
   direct MK8D boot regression during this step). Subsystem B
   (svc_lock) is naturally covered because `ArbitrateLock` /
   `ArbitrateUnlock` just forward into condvar.

### Still pending

5. ⏳ **Step 2 — address arbiter** — `k_address_arbiter.rs` is a full
   rewrite, not a small scope change. Current Rust impl uses a
   `HashMap<u64, parking_lot::Condvar>` and blocks host threads
   directly via `cv.wait_timeout(...)`. Upstream uses the
   scheduler-lock + intrusive tree + `BeginWait` / `EndWait` pattern
   (see `zuyu/src/core/hle/kernel/k_address_arbiter.cpp`). MK8D boot
   doesn't exercise the arbiter path heavily at current progress, so
   this is lower priority than subsystem D below.
6. ⏳ **Step 2 — wait-sync path (`svc_synchronization.rs`)** — mirror
   the same "kernel::scheduler_lock() + drop lock before fiber-wait"
   pattern used for condvar.
7. ⏳ **Step 4 — cleanup** — remove now-dead `TrackedMutex` module +
   `ProcessLockTracker` (except sites that still reference them for
   SIGUSR1 diagnostics outside the refactored subsystems).
